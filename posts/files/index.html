1. The Real-World Problem

You're building a product search engine. A user asks: "Show me laptops similar to this one, under $1,000, from Apple or Dell."

This query has two distinct parts:

  • Vector similarity: Find products with similar embeddings to the query
  • Metadata filtering: Only return items matching specific criteria

Modern vector databases use HNSW (Hierarchical Navigable Small World) graphs for fast similarity search. But HNSW was designed for unfiltered search. When you add filters, things break in subtle and dangerous ways.

This tutorial explains exactly why, and how Qdrant's filterable HNSW algorithm solves it.

2. Graph Theory Fundamentals

Before diving into HNSW, we need to understand what graphs are and why connectivity matters.

What is a Graph?

A graph is a mathematical structure consisting of nodes (points) connected by edges (lines). Think of it like a subway map: stations are nodes, tracks are edges.

Term Definition Subway Analogy
Node A data point in the graph A station
Edge A connection between two nodes A track between stations
Path A sequence of edges connecting nodes Your route from A to F
Connected Any node reachable from any other You can get anywhere

Why Connectivity Matters

A graph is connected if you can reach any node from any other by following edges. If not, the graph has disconnected components—isolated islands with no path between them.

When node B is removed, nodes A and C become disconnected—there's no path between them.

This distinction becomes critical when we start removing nodes from a graph.

3. How HNSW Works

HNSW is a graph-based algorithm for approximate nearest neighbor search. Understanding it requires two key intuitions.

Intuition 1: The Navigation Graph

Imagine you're dropped in a foreign city and need to find the best coffee shop. You have two strategies:

Strategy A (Brute Force): Visit every coffee shop, rate each one, pick the best. Guaranteed to find the best, but takes forever—O(n) time.

Strategy B (Ask Locals): Ask someone nearby for directions. Go there, ask again. Keep following recommendations until you converge on the answer. Fast—O(log n) time—but might miss the absolute best (approximate).

HNSW implements Strategy B. It builds a graph where each data point connects to its most similar neighbors. To search, you start somewhere and "hop" through the graph, always moving toward more similar items.

Intuition 2: The Hierarchical Layers

The "H" in HNSW stands for Hierarchical. The graph has multiple layers, like zooming in on a map:

HNSW uses multiple layers: top layers for coarse navigation, bottom layers for fine-grained search.

You start at the top (coarse navigation) and descend (fine navigation). This hierarchical structure is what makes HNSW fast—you don't search the entire bottom layer; you descend from above, already in the right neighborhood.

The m Parameter

The m parameter controls how many edges each node has. Higher m means better search quality and more robust connectivity, but also higher memory usage and slower index construction. Typical values range from 8 to 64. Qdrant defaults to m=16.

4. The Filtering Problem

Now we reach the core challenge. Vector search rarely happens in isolation—users want filtered results.

The Two Naive Approaches (And Why They Fail)

⚠️ Approach 1: Post-Filtering

Search HNSW graph → get top-K results → filter out non-matching items → return filtered list.

Problem: What if only 3 of those 100 items match your filter? You'll return poor results—the truly relevant items weren't in the initial top-K.

⚠️ Approach 2: Pre-Filtering

Filter dataset → keep only matching items → search within filtered subset.

Problem: You can't just "search within the filtered subset" using HNSW—the graph was built on ALL data. When you remove nodes, you might break the graph.

Both naive approaches fail: post-filtering misses results; pre-filtering breaks graph connectivity.

This is the fundamental problem: the HNSW graph's connectivity depends on nodes that might be filtered out.

5. Understanding the Damage (Percolation Theory)

How bad does filtering get? Percolation theory gives us a precise answer.

The Critical Threshold

Percolation theory studies when networks fall apart. The key result is:

pc = 1 / ⟨k⟩

Where:

  • pc = critical threshold (minimum fraction of nodes to keep the graph connected)
  • ⟨k⟩ = average number of edges per node

With m=16 edges per node, you can randomly remove ~94% of nodes and the graph likely stays connected.

This sounds great—until you realize that:

  1. Filters often remove more than 94% of data
  2. Filter removal isn't random—it's systematic

When a filter matches only 0.5% of your data, you're almost certainly below the threshold. The remaining nodes form disconnected islands—and HNSW can only find results on whichever island you started.

6. Qdrant's Solution: Pre-Built Subgraphs

Qdrant's insight is elegant: don't rely on the main graph for filtered queries—pre-build connected subgraphs for each filter value.

The Core Algorithm

For a categorical field like category with values [laptop, phone, tablet]:

  1. Build a mini-HNSW graph connecting only laptops to each other
  2. Build a mini-HNSW graph connecting only phones to each other
  3. Build a mini-HNSW graph connecting only tablets to each other
  4. Merge all these edges into the main graph

Now, when you filter for "laptops only," there's a guaranteed connected path through all laptops—because you explicitly built one!

By building category-specific edges and merging them, Qdrant guarantees connectivity within each category.

Space Overhead Analysis

A key concern: doesn't this explode memory usage?

✓ Surprisingly efficient

The total edges increase by at most 2×, regardless of how many categories exist. Each node appears in exactly one category subgraph, so each node gains at most m additional edges. Overhead is constant regardless of category count.

7. The Query Planner

Qdrant doesn't blindly use filterable HNSW for every query. A query planner selects the optimal strategy at runtime based on filter selectivity.

The query planner adaptively chooses between standard HNSW, filterable HNSW, and full scan.

Filter Selectivity Strategy Reasoning
High (70%+) Standard HNSW Graph is still well-connected
Medium (5-70%) Filterable HNSW Use pre-built subgraph edges
Low (<5%) Full scan Candidate set is tiny

The full_scan_threshold parameter controls when to switch from HNSW to full scan. When the filtered candidate count falls below the threshold, Qdrant switches to brute-force scanning—which is actually faster for tiny result sets.

8. Handling Different Filter Types

Not all filters are categorical. Qdrant handles different types with different strategies.

Categorical Filters (Optimal Case)

Direct subgraph construction. Each category value gets its own connected component. This is the optimal case for filterable HNSW.

Numerical Range Filters (Approximated)

For ranges like price < 1000, Qdrant discretizes into buckets. Neighboring buckets are connected to each other, enabling range traversal. Edge cases near bucket boundaries require post-filtering.

Geographical Filters

Location-based filters use geohash encoding. Points within the same geohash prefix form a subgraph, with edges to neighboring prefixes for smooth geographical traversal.

Multi-Field Filters (Challenging Case)

Complex filters combining multiple fields don't have pre-built subgraphs. Qdrant handles this by using individual field indexes, taking intersection of candidate sets, and applying additional post-filtering as needed.

9. Implementation Requirements

⚠️ Critical: Payload Indexes Must Be Created Explicitly

Qdrant does not automatically index payload fields. You must create indexes explicitly, ideally BEFORE uploading data.

# Create the collection
client.create_collection(
    collection_name="products",
    vectors_config=VectorParams(size=384, distance=Distance.COSINE),
)

# MUST create payload indexes for filtered fields
client.create_payload_index(
    collection_name="products",
    field_name="category",
    field_schema=PayloadSchemaType.KEYWORD,
)

client.create_payload_index(
    collection_name="products",
    field_name="price",
    field_schema=PayloadSchemaType.FLOAT,
)

Without these indexes, filtered queries fall back to post-filtering—losing all the benefits of filterable HNSW.

Timing Matters

Best Practice: Create indexes BEFORE uploading data. This ensures the HNSW graph is built with filter-aware edges from the start.

If you create indexes after data upload, the HNSW graph won't have the subgraph edges. You'll need to rebuild the index by updating the HNSW configuration—an expensive operation for large collections.

10. Limitations & Edge Cases

No algorithm is perfect. Here's an honest assessment of filterable HNSW's limitations.

Six key limitations to be aware of when using filterable HNSW.

1. Non-Indexed Fields

Filters on non-indexed fields fall back to post-filtering. You must know your query patterns upfront.

2. Categorical Bias

Works optimally for discrete categories. Continuous values require approximation via bucketing with boundary filtering overhead.

3. Complex Multi-Field Filters

No pre-built subgraph for arbitrary combinations. Performance degrades as filter complexity increases.

4. Index Timing

Indexes created after HNSW is built won't have subgraph edges—requiring expensive rebuild.

5. Filter-Vector Correlation

Percolation theory assumes random removal. If filters correlate with vector semantics (e.g., all laptops cluster together), theoretical guarantees weaken.

6. Memory Overhead

Each indexed field adds storage for the payload index plus up to 2× HNSW edges. Index only frequently filtered fields.

11. Summary & Best Practices

The Key Insights

  1. The problem: HNSW graphs break when filtering removes nodes that served as bridges
  2. The solution: Pre-build connected subgraphs for each filter value, merge into main graph
  3. The guarantee: Filtered queries traverse the subgraph edges, maintaining connectivity
  4. The trade-off: 2× edge overhead, requires upfront index planning

Best Practices Checklist

  • ☑️ Create payload indexes BEFORE uploading data
  • ☑️ Index only frequently-filtered fields to minimize overhead
  • ☑️ Use categorical filters when possible (they're optimal)
  • ☑️ Benchmark with realistic filter selectivities
  • ☑️ Monitor memory usage as you add indexes
  • ☑️ Consider full_scan_threshold tuning for your workload
